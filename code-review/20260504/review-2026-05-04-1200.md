# 静的コードレビュー

**日時**: 2026-05-04  
**レビュー対象**: 取引管理システム（全ソースファイル）  
**観点**:
1. 部品間の実装整合性
2. 要件定義書との矛盾・要件定義の修正要否

---

## 1. 部品間の実装整合性

### 🔴 重大

#### 1-1. `discount` の意味が quotation.js と purchaseOrder.js で異なる

| モジュール | discount の扱い | 計算式 |
|---|---|---|
| `quotation.js` | **絶対額**（円） | `unitPrice × quantity − discount` |
| `purchaseOrder.js` | **割合**（0〜1） | `quantity × unitPrice × (1 − discount)` |

両モジュールのテストはそれぞれの解釈で通過するが、受注から発注に明細を引き継ぐ際に discount の値が異なる意味で解釈される。見積・受注の discount（円額）が発注の discount（割合）として引き継がれると計算が破綻する。

**修正案**: どちらか一方の解釈に統一する（絶対額が業務的に自然）。

---

#### 1-2. 初期サンプルデータの整合性破綻

`app.js` の `orders[1]`（ORD-00002）と参照先の見積の customerId が不一致。

```js
// ORD-00002
quotationCode: "QUO-00004",  // QUO-00004 の customerId は "CUS-005"
customerId:    "CUS-003",    // ← 不一致
```

見積から受注への引き継ぎの整合性が壊れており、E2Eテストや画面表示で意図しない挙動が発生しうる。

---

### 🟡 中程度

#### 1-3. `findOrderByCode` / `findPurchaseOrderByCode` の戻り値が他の `find*` 関数と不統一

| 関数 | 見つからない場合 |
|---|---|
| `findCustomerByCode` | `null` |
| `findSupplierByCode` | `null` |
| `findProjectByCode` | `null` |
| `findQuotationByCode` | `null` |
| `findUserById` | `null` |
| `findOrderByCode` | **`undefined`** |
| `findPurchaseOrderByCode` | **`undefined`** |

テストも `toBeNull()` と `toBeUndefined()` が混在しており、呼び出し側での存在確認が不統一になりやすい。

**修正案**: `findOrderByCode` と `findPurchaseOrderByCode` に `|| null` を追加して統一する。

---

#### 1-4. `createUser` の権限定義と `app.js` のハードコードユーザーで権限が異なる

`user.js` の `ADMIN_PERMISSIONS` には `purchase-order:edit` が**含まれない**が、`app.js` のadminユーザーには含まれている。`createUser` でシステム管理者を作成した場合と、初期データのadminユーザーで権限が異なり、ユーザー管理画面から作成したシステム管理者は発注編集ができない。

---

#### 1-5. `product.js` が `customer.js` の内部関数に依存している

```js
// product.js
import { generateCode } from './customer.js';
```

`generateCode` は顧客・仕入先向けのコード採番関数として定義されているが、商品コード生成が顧客モジュールに依存する設計は、責務の境界が不明確。共通ユーティリティとして分離するか、`product.js` 内に独立した採番関数を持つべき。

---

#### 1-6. 商品マスタの `unitPrice` が文字列型で格納されている

```js
// app.js
{ code: "PRD-001", name: "サーバー保守サービス", unitPrice: "50000", ... }
```

`createProduct` は型変換を行わない。見積明細への商品マスタ参照時に `unitPrice` が文字列のままだと計算結果が文字列連結になる（例：`"50000" * 12 = 600000` は偶然数値として動くが、`"50000" + something` は不具合になる）。

**修正案**: 初期データの `unitPrice` を数値型にする、または `createProduct` で `Number()` 変換を行う。

---

#### 1-7. `screens` 配列に S-07（納品・検収画面）が欠落

```js
// app.js の screens 配列: S-07 が存在しない
// S-02, S-03, S-04, S-05, S-06, S-08, S-09, S-10, S-11, S-12, S-13, S-14
```

メニューから納品・検収画面へ到達する導線がなく、`delivery.js` のビジネスロジックをUIから利用できない状態。

---

#### 1-8. 発注ステータスに定義外の値が存在する

`app.js` の初期データに `"承認済・発注待ち"` というステータスが使われているが、`purchaseOrder.js` の `submitPurchaseOrderApproval` が設定するのは `'承認依頼中'`。コード内に存在しないステータス値がデータに混入している。

---

### 🟢 軽微

#### 1-9. コード採番の桁数が種別により不統一

| 種別 | 桁数 | 例 |
|---|---|---|
| 顧客(CUS)・仕入先(SUP)・商品(PRD) | 3桁 | CUS-001 |
| 案件(PJ)・見積(QUO)・受注(ORD)・発注(POD)・納品(DLV)・請求(INV) | 5桁 | PJ-00001 |

業務上の意味は通じるが、コード体系の一貫性がなく、将来的に3桁を超えた場合に採番フォーマットが崩れる。

---

#### 1-10. `invoice.js` の `findBillableOrders` が分割請求に対応していない

```js
// 一度でも invoice が作成された orderCode は除外される
const invoicedOrderCodes = invoices.map(inv => inv.orderCode);
return orders.filter(order => order.billingTarget && !invoicedOrderCodes.includes(order.code));
```

同一受注に対する2回目以降の請求（分割請求）ができない実装になっている。

---

## 2. 要件定義書との矛盾

### 🔴 重大

#### 2-1. 見積ステータスに要件定義に存在しない「却下」が使われている

- **要件定義 7.3**: 見積の状態 → `下書き` / `承認依頼中` / `承認済み` / `失注` / `取消`
- **実装 `rejectQuotation`**: status を `'却下'` に設定

承認フローで「却下」は承認操作の結果として適切だが、見積伝票のステータスとしては要件定義に定義がない。

**要件定義への追加を推奨**: 7.3 に「却下（承認者による否認）」を追加するか、見積ステータスを「取消」に変更する旨を明確化する。

---

#### 2-2. S-07（納品・検収登録画面）がメニューから欠落

- **要件定義 8.1**: S-07「納品、検収登録画面」が定義されている（主な利用者：購買部門）
- **実装**: `screens` 配列に S-07 が含まれず、UIから到達不可

---

#### 2-3. 操作ログ・監査証跡が全エンティティで未実装

- **要件定義 9.3 / F-17**: 作成者・更新者・作成日時・更新日時の保持、重要操作の履歴保存
- **実装**: 全モジュール（customer.js, project.js, quotation.js 等）の `create*` 関数に `createdBy`・`createdAt`・`updatedBy`・`updatedAt` フィールドが存在しない

優先度「高」（F-17）の機能が全く実装されていない。

---

### 🟡 中程度

#### 2-4. 案件に「担当者」と「予定金額」フィールドがない

- **要件定義 7.2**: 案件は「担当者、予定金額」を管理できること
- **実装 `createProject`**: `managerId`（担当者）・`estimatedAmount`（予定金額）フィールドが存在しない

---

#### 2-5. 受注承認の見積条件一致チェックが未実装

- **要件定義 6.2.2**: 「紐づけした見積りと条件が異なる場合は受注申請を回付することができないように制御する」
- **実装**: `createOrderFromQuotation` は見積コードを引き継ぐが、金額・明細等の条件一致チェックは実装されていない

---

#### 2-6. 段階的承認フロー（営業部長→社長決裁）が未実装

- **要件定義 6.1**: 見積は「営業部長の承認」が必要。値引き条件次第では「社長の承認」が必要
- **実装**: 承認は `approval:act` 権限の有無のみで判定。役職（部長・社長）による段階的承認ロジックがない

---

#### 2-7. 認証連続失敗カウントとアカウントロックが未実装

- **要件定義 8.2.1（S-01）**: 「連続失敗を記録できること」「アカウントロック状態を通知できること」
- **実装**: ログイン失敗のエラーメッセージ表示のみ。カウント・ロック機能なし

---

#### 2-8. 支払モジュール（payment.js）のビジネスロジックが未実装

- **要件定義 7.9 / F-10**: 支払依頼作成・承認・実績登録・未払管理
- **実装**: `payment.js` が存在せず、ビジネスロジックがない。UIの `payment:view`/`payment:edit` 権限のみが定義されている

---

#### 2-9. 入金モジュール（receipt.js）のビジネスロジックが未実装

- **要件定義 7.8 / F-09**: 入金登録・消込・未収管理
- **実装**: `receipt.js` が存在しない

---

#### 2-10. CSV出力が未実装

- **要件定義 7.12 / F-15**: 「CSV出力に対応すること」（優先度「中」）
- **実装**: 対応なし

---

### 🟢 軽微

#### 2-11. 帳票出力が一部のみ実装

- **要件定義 8.3**: 見積書・受注確認書・発注書・納品書・請求書・支払通知書・売上集計表・粗利集計表・未収一覧・未払一覧
- **実装**: 見積書（`buildQuotationPrintHtml`）・発注書（`buildPurchaseOrderPrintHtml`）のみ

---

#### 2-12. パスワードが平文でソースに記載

- **要件定義 12.3（セキュリティ要件）**: 認証・暗号化要件あり
- **実装**: `app.js` にパスワードが平文で記載（`password: "admin123"` 等）

現フェーズはプロトタイプであるため許容しうるが、本番移行時に必ず修正が必要な箇所として明示しておく。

---

## 3. 要件定義書への修正推奨事項

以下は実装から見て要件定義書を更新すべき箇所。

| 番号 | 箇所 | 修正内容 |
|---|---|---|
| R-1 | 7.3 見積管理要件 | 見積ステータスに「**却下**」を追加する（承認者による否認）か、実装で「取消」に変更する旨を明示 |
| R-2 | 15.1 業務上の制約（初期リリース） | 段階的承認フロー（部長→社長）・役職別承認制御は初期リリース対象外であることを明示 |
| R-3 | 15.2 システム上の制約 | 操作ログ・監査証跡は初期リリース対象外か否かを明示（現在F-17は優先度「高」だが未実装） |
| R-4 | 7.2 案件管理要件 | 案件の「**担当者**」フィールドの型（ユーザIDか名前か）と、「**予定金額**」フィールドの定義を補足 |
| R-5 | 8.1 画面一覧（S-07） | 納品・検収画面の主な利用者が「購買部門」のみになっているが、業務フロー図（6.4.1）では「営業担当者」も作業報告を入力する。S-07の利用者を再確認する |

## 4. まとめ

| カテゴリ | 件数 | 主な内容 |
|---|---|---|
| 🔴 重大（部品間） | 2件 | discountの意味相違、初期データの整合性破綻 |
| 🟡 中程度（部品間） | 6件 | find*の戻り値不統一、権限差異、product→customer依存 等 |
| 🟢 軽微（部品間） | 2件 | 桁数不統一、分割請求対応漏れ |
| 🔴 重大（要件定義との矛盾） | 3件 | 「却下」ステータス未定義、S-07欠落、操作ログ未実装 |
| 🟡 中程度（要件定義との矛盾） | 6件 | 承認フロー、支払・入金モジュール未実装 等 |
| 🟢 軽微（要件定義との矛盾） | 2件 | 帳票出力の不完全、パスワード平文 |
| 要件定義修正推奨 | 5件 | 「却下」ステータスの明示、初期リリース制約の明確化 等 |

最優先で対処すべきは **discount の意味統一**（1-1）と **`findOrderByCode` の戻り値統一**（1-3）です。これらは既存テストが通過しているにもかかわらず、実際の業務処理で誤った計算や null チェック漏れを引き起こすリスクがあります。
