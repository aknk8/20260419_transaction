## 前回指摘の対応状況

| ID     | 重要度 | 内容                             | 対応状況     |
|--------|--------|----------------------------------|--------------|
| VA-01  | P1     | API が認証のみで認可を行っていない | **対応済み** |
| VA-02  | P1     | JWT 秘密鍵未設定でも起動できる    | **対応済み** |
| VA-03  | P2     | 通知既読化の所有者チェックなし    | **対応済み** |
| VA-04  | P2     | Mass Assignment / 入力スキーマ不足 | **一部対応** |
| VA-05  | P2     | ログイン試行への防御が弱い        | **対応済み** |
| VA-06  | P2     | JWT をサーバー側で失効できない     | **対応済み** |
| VA-07  | P2     | CSRF 対策が SameSite 依存のみ     | **対応済み** |
| VA-08  | P3     | フロント権限判定がローカル定義に依存 | **一部対応** |
| VA-09  | P3     | innerHTML による DOM XSS リスク   | **未対応**   |

### VA-01 対応確認 ✓

`server/plugins/authorization.js` に `fastify.requirePermission()` が実装され、全ルートに適用されている。  
`server/routes/` 配下の更新系 API に `preHandler: [fastify.authenticate, fastify.requirePermission(...)]` が付与されており、401/403 テストも存在する。

### VA-02 対応確認 ✓

`server/startupGuards.js` が実装され、`NODE_ENV === 'production'` 時に `JWT_SECRET` 未設定のまま起動するとプロセスが終了する。  
`server/app.js` のフォールバック文字列は開発・テスト用途のみ。

### VA-03 対応確認 ✓

`server/routes/notifications.js:13` で `markAsRead(req.params.id, req.user.id)` を呼ぶよう変更済み。  
`server/services/notificationService.js:31` で `notification.recipientId !== requestingUserId` の場合に 403 または 404 を返す。

### VA-04 一部対応

`server/routes/users.js` に `additionalProperties: false` を含む Fastify スキーマが定義され、ユーザ更新の Mass Assignment は防止された。  
ただし `customers.js`, `products.js`, `quotations.js`, `orders.js`, `purchaseOrders.js`, `invoices.js`, `payments.js` などにはリクエストボディスキーマが未定義のまま。これらは引き続き指摘継続（下記 VA-04 参照）。

### VA-05 対応確認 ✓

`server/routes/auth.js` の `/login` に専用 rate limit (本番 5 req/min) が実装済み。アカウントロック (5 回失敗で 30 分) も `authService` で管理されている。

### VA-06 対応確認 ✓

JWT に `jti` を付与し、`server/repositories/sessionRepository.js` (またはインメモリ Map) でセッション管理を行うよう実装済み。ログアウト時に jti を失効させ、再利用を検出する。  
なお、セッションストアが現状インメモリである点はプロダクション前に永続化が必要（後述 VA-10）。

### VA-07 対応確認 ✓

`server/plugins/csrf.js` が実装され、POST / PUT / PATCH / DELETE に対して Origin ヘッダ検証が行われる。`app.js` に登録済みで、`auth.test.js` にテストが存在する。

### VA-08 一部対応

`app.js:1563-1566` にて「サーバーが `permissions` を返す場合はそれを優先」するロジックが追加された（コメントで VA-08 と明記）。  
ただし `app.js:5380` に旧コード（`localUser ? localUser.permissions : []`）が残存しており、/api/auth/me のレスポンスに `permissions` が含まれていてもローカル定義のみを使う実装パスが存在する。また `app.js:33-128` のローカルユーザ定義は削除されていない。

### VA-09 未対応

`app.js` の UI 描画は依然として大きな HTML 文字列連結 + `innerHTML` が中心。`escapeHtml` の適用は多いが、実装面積が大きく escape 漏れが発生しやすい構造のまま。

## 新規診断結果

### VA-04 [P2] 継続：マスタ系ルートの入力スキーマ未整備（Mass Assignment）

該当箇所:

- `server/routes/customers.js`
- `server/routes/products.js`
- `server/routes/quotations.js`
- `server/routes/orders.js`
- `server/routes/purchaseOrders.js`
- `server/routes/invoices.js`
- `server/routes/payments.js`
- `server/routes/suppliers.js`
- `server/routes/projects.js`

内容:

上記ルートはリクエストボディの Fastify スキーマ（または同等の Zod バリデーション）が未定義のまま。意図しないフィールドがサービス層まで到達し保存される可能性がある。`users.js` の対応が参考実装として利用できる。

推奨対応:

- 各ルートに `schema: { body: { type: 'object', additionalProperties: false, properties: {...}, required: [...] } }` を追加する。
- または Zod スキーマを用いて `body = schema.parse(req.body)` を行い、許可フィールドのみを取り出す。

### VA-08 [P3] 継続：フロントエンド権限のローカル定義残存

該当箇所:

- `app.js:5380`（旧コード残存 — サーバー権限を使わないパス）
- `app.js:33`, `68`, `93`, `115`, `128`（ローカルユーザ定義）

内容:

`app.js:5380` では `localUser ? localUser.permissions : []` と書かれており、`/api/auth/me` が `permissions` を返していてもそれを参照しない。サーバー権限とフロント権限が一致しないリスクが残る。

推奨対応:

- `app.js:5380` を `app.js:1564-1566` と同様のパターン（サーバー権限優先）に統一する。
- 最終的にローカルユーザ定義 (`app.js:33-128` の `users` 配列) から `permissions` 項目を削除し、サーバー権限のみを正とする。

### VA-09 [P3] 継続：innerHTML による DOM XSS リスク

該当箇所:

- `app.js` 全体（`innerHTML` を使う箇所が多数）

内容:

前回指摘から変更なし。`escapeHtml` の適用漏れが将来的に発生するリスクは継続している。

推奨対応:

前回と同じ。DOM API またはテンプレートラッパーへの移行を段階的に進める。

### VA-10 [P2] 新規：セッション・リフレッシュトークンのインメモリ保存

該当箇所:

- `server/repositories/sessionRepository.js`（インメモリ Map）

内容:

セッション管理 (jti) とリフレッシュトークンがインメモリに保存されている。アプリケーション再起動でセッションが全失効し、複数インスタンス構成では同一セッションが他インスタンスで認証されない。

影響:

本番環境でのローリングデプロイや障害によるプロセス再起動時にユーザが強制ログアウトされる。水平スケール時にセッション共有が不可能。

推奨対応:

- Redis または DB テーブルによる永続セッションストアを実装する。
- マルチインスタンス対応のためにセッション共有設計を先行して決定する。

## 悪意あるコード・出所不明ライブラリの診断

実施内容:

- `package.json` の全依存を確認
- `package-lock.json` の `resolved` URL を確認
- `file:`, `git+`, `github:`, `http://` 依存の有無を確認
- `postinstall`, `preinstall`, `prepare` スクリプトの有無を確認
- `eval()`, `new Function()`, `child_process.exec()` のユーザ入力への渡し、ネットワーク接続の静的検索
- 隠しファイル・バイナリファイルの確認

結果:

- lockfile 上のパッケージ数は 169 件。
- `package-lock.json` の `resolved` はすべて npm 公式レジストリ (`https://registry.npmjs.org/`) 由来。
- `file:`, `git+`, `github:`, `http://` 依存は検出されなかった。
- 不審なライフサイクルスクリプトなし。
- `hasInstallScript` は `esbuild`, `fsevents`, `playwright/node_modules/fsevents` のみ。いずれもネイティブバイナリ選択または optional dependency 用途として一般的であり、不審とは判断しない。
- ソースコード中に `eval()` / `new Function()` / `spawn()` のユーザ入力渡しなし。
- アプリ本体に `child_process.exec()` / `spawn()` / `fork()` の利用なし。
- 外部の不明 API への送信処理なし。外部 URL は localhost、staging/production の例示 URL、npm registry、GitHub Actions 上のヘルスチェック用途が中心。
- ハードコードされた API キー・資格情報なし（開発用プレースホルダと `.env.*` のローカル設定のみ）。
- `.env.staging` / `.env.production` は `.gitignore` で除外されており、リポジトリに含まれていない。
- 依存ライブラリはすべて well-known の公開パッケージ（npm 公式レジストリ）。
- 出所不明・悪意あるコードは検出されなかった。

補足:

- `scripts/deploy-staging.sh` は SSH によるステージングデプロイを行う。用途は妥当だが、`StrictHostKeyChecking=no` はホスト鍵検証を弱めるため、運用リスクとしては改善余地がある。
- `scripts/init-tls.sh` は certbot コンテナで Let's Encrypt 証明書を取得するためのスクリプトであり、不審な挙動は見られない。
- `scripts/seed-staging.js` はステージング用 SQL を標準出力へ生成するだけであり、不審な外部通信はない。

## 推奨する自動テスト追加

- マスタ系 POST / PATCH に未定義フィールドを含めたとき 400 になること。
- `/api/auth/me` のレスポンスに `permissions` が含まれること。
- セッション失効後（ログアウト後）に jti を使って API 呼び出しが 401 になること。
- アプリ再起動後に再ログイン不要でセッションが維持されること（永続化後）。
