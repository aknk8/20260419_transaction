# レビュー・テスト手法集

このファイルは、本プロジェクトで実施したレビュー・テスト手法を、他のプロジェクトや他のフェーズでも再利用できるよう手順として記録したものです。

---

##  1：静的コードレビュー

### 目的

実装済みコードを静的に読み、以下の2つの観点で問題を洗い出す。

1. **部品間の実装整合性**：モジュール間でロジックの解釈や命名が一致しているか
2. **要件定義との矛盾**：実装が要件定義書の記述と食い違っていないか（要件定義自体の更新が必要な箇所も含む）

### 担当

テックリード（主導）、バックエンドエンジニア（協力）

### 手順

1. 要件定義書（またはそれに相当する文書）を通読し、主要な仕様・ルールを把握する
2. 各モジュール（src/*.js 等）のソースコードとそのテストコードを順に読む
3. 以下の観点でメモを取る
   - 同じ概念（例：discount の扱い）が複数モジュールに跨るとき、解釈が一致しているか
   - 関数の戻り値の型（null vs undefined など）が他モジュールの呼び出し側と整合しているか
   - 要件定義のステータス定義・フロー・ルールと実装内容が一致しているか
4. 発見事項を重大度（致命的・高・中・低）付きで記録する
5. 要件定義を修正すべき箇所があれば「要件定義更新推奨」として別項に記載する

### 成果物

`/code-review/review-yyyy-mm-dd-hhmm.md`

### テンプレート構成

```
# コードレビュー
## Part 1: 部品間の実装整合性
### 発見 C-XX: タイトル（重大度）
- 対象: ファイル名・行番号
- 問題: 具体的な差異
- 修正方針
## Part 2: 要件定義との矛盾
### 発見 R-XX: タイトル（重大度）
- 該当要件: セクション番号
- 問題: 矛盾の内容
- 修正方針
## 要件定義更新推奨
```

### 注意点

- テストコードも確認対象に含める（実装の意図が読み取れる）
- `app.js` のような統合ファイルは初期データ・ルーティング・レンダリングを分けて読む
- ステータス値・フォーマット・採番規則など「暗黙ルール」が不整合の温床になりやすい

##  2：探索テスト（SBTM）

### 目的

自動テストでは発見しにくい「動作するが正しくない」挙動・UX上の問題・画面間の不整合を、人間の探索的操作によって発見する。

### 手法

SBTM（Session-Based Test Management）：チャーター（探索目的）を事前に定め、セッション単位でテストを実施・記録する。

### 担当

| フェーズ | 担当 |
|---------|------|
| 計画・チャーター作成 | QAエンジニア（全チームでレビュー） |
| 実施可否判断 | テックリード |
| テスト実施・記録 | QAエンジニア |
| 結果レビュー | プロダクトマネジャー |

### 手順

#### フェーズ 1：計画（explore-testing-plan.md の作成）

1. テスト対象の画面・機能を洗い出す（要件定義・実装計画を参照）
2. チャーター（探索目的）を業務フロー別に設計する
   - 例：「ET-01 ログイン・権限」「ET-02 見積フロー」など
3. 各チャーターに「探索する内容」「気になること」「使用するデータ」を記述する
4. テックリードが「現時点でテスト開始が妥当か」を判断する（未実装画面の多さ等を確認）
5. 実施順序と未実施チャーター（延期理由）を明記する

#### フェーズ 2：テスト実施（Playwright スクリプト作成・実行）

1. 各チャーターに対して Playwright スクリプトを作成する
2. スクリプトは「アサーションで失敗させる」より「観察して console.log で出力する」スタイルを優先する（探索が目的のため）
3. アプリ固有の DOM 構造を事前確認する（本プロジェクトでは `<table>` ではなく `div.data-table-body-row` を使用）
4. スクリプト実行後、出力ログから発見事項を整理する

#### フェーズ 3：結果記録

`/code-review/explore-test-result_yyyymmdd.md` に以下を記録する。

```
## サマリー（チャーター別 Pass/Fail/バグ数）
## 検出バグ一覧（重要度・タイトル・チャーター）
## チャーター別詳細
  - テスト項目と結果（表形式）
  - バグの再現手順・期待値・実際値・推定原因
  - 正常動作の確認内容
## 延期チャーターと理由
## 推奨アクション
```

### チャーター設計のパターン

| パターン | 内容 |
|---------|------|
| 認証・権限 | ユーザ種別ごとのメニュー表示・操作制限・ログアウト後のセキュリティ |
| 基本フロー | 主要業務の一連の操作（作成→承認→確定→出力） |
| マスタ管理 | 一覧表示件数・ページング・検索・新規登録の自動採番 |
| バリデーション | 必須チェック・重複チェック・エラー後の再入力 |
| 権限横断 | 閲覧専用ユーザによる書き込み操作の制限確認 |

### 注意点

- 「静的レビューで発見済みのバグ」はテスト前に修正または記録しておく（探索時間を有効活用するため）
- スクリプトが失敗した場合、「アプリのバグ」と「スクリプトのセレクター誤り」を切り分ける
- 未実装画面（スタブ）はテスト対象外として明記し、実装後に再実施する

##  3：デザイン整合性レビュー

### 目的

全画面を横断的に調査し、UI部品（ボタン・テーブル・フォーム・ステータスバッジ等）のクラス名・ラベル・構造が画面間で一致しているかを確認する。

### 担当

UXデザイナー（主導）、フロントエンドエンジニア（協力）

### 手順

1. `styles.css`（または相当するスタイル定義ファイル）を通読し、定義済みクラスを把握する
2. 全画面レンダリング関数（`renderXxxScreen`・`buildXxxHtml` 等）を横断して読む
3. 以下の観点で不整合を記録する

| 確認観点 | チェック内容 |
|---------|-----------|
| CSS クラスの実在 | 使用クラスが styles.css に定義されているか（未定義 → スタイル崩壊） |
| ステータスバッジ | `status` / `status-badge` 等、同じ目的に複数クラスが混在していないか |
| ボタンクラス | `button-sm` / `button-xs` 等、サイズ修飾子が統一されているか |
| 詳細行要素 | `detail-row` / `detail-field` / `detail-item` 等、コンテナクラスが統一されているか |
| アクションボタン配置 | `toolbar` / `panel-actions` 等、コンテナクラスが統一されているか |
| ボタンラベル | 「一覧へ戻る」/「一覧に戻る」等、同一機能のテキストが統一されているか |
| フォーム送信ボタン | 「保存」/「登録する」/「受注登録」等、ラベルパターンが統一されているか |
| 新規登録ボタン | 「新規登録」/「新規〇〇登録」/「新規〇〇作成」等が統一されているか |

4. 発見事項を重大度（致命的・高・中・低）で分類する
   - 🔴 致命的：使用クラスが未定義（スタイル崩壊確定）
   - 🟠 高：同じ目的で異なるクラス名（見た目が変わる）
   - 🟡 中：テキスト・コピーの表記ゆれ
   - 🟢 低：意味的なゆれ（機能・見た目への影響小）
5. フロントエンドエンジニアへの申し送りとして「未定義クラス一覧」を別項に記載する

### 成果物

`/code-review/design-consistency-review-yyyy-mm-dd.md`

### 注意点

- 「定義済みクラスを使っている画面」と「独自クラスを使っている画面」が混在しやすい
- 同一ファイル内にヘルパー関数が重複定義されている場合（`fieldHtml` 等）も不整合の温床
- スタイル定義とHTMLの両方から問題を見つける（片方だけでは見落としが生じる）

##  4：ワークフロー要件の実装可能性検証

### 目的

要件定義書のワークフロー関連記述が、エンジニアが実装着手できる粒度に達しているかを検証する。「何を作るか」だけでなく「どう動くか」が記述されているかを確認する。

### 担当

| 観点 | 担当 |
|------|------|
| 業務要件の完全性 | プロダクトマネジャー |
| 技術実装可能性 | テックリード |
| データモデル・状態遷移 | バックエンドエンジニア |
| 画面遷移・UI | フロントエンドエンジニア |
| フロー図の抜け漏れ | UXデザイナー |

### 手順

1. 要件定義書のワークフロー関連セクション（業務フロー・機能要件・画面要件・権限要件）を通読する
2. 各担当が自分の観点で「実装するために知らなければならない情報」を書き出す
3. その情報が要件定義書に記載されているか確認し、記載がなければ「問題」として記録する
4. 発見事項を重大度（実装不可・実装困難・要補足）で分類する

### 確認チェックリスト

#### プロダクトマネジャー観点

- [ ] 承認条件（社長決裁が必要になる条件など）の型と評価対象が定義されているか
- [ ] 差戻しと却下の業務的な違いと後続フローが定義されているか
- [ ] 条件一致チェックの対象項目と判定ルールが定義されているか
- [ ] 例外処理（受注取消・一部入金等）の後続業務が定義されているか
- [ ] 全伝票のステータス一覧と有効な遷移パスが定義されているか

#### テックリード観点

- [ ] 承認者の特定方法（役職＋部門マッチ・静的設定・動的マスタ管理）が決定されているか
- [ ] 複数候補がいる場合の承認者決定ルールが定義されているか
- [ ] 承認フローのステップ数が動的か静的かが決定されているか（ハードコード可否）
- [ ] 通知のトリガー・送達先・手段が定義されているか
- [ ] 添付ファイルの必須/任意とチェックタイミングが定義されているか

#### バックエンドエンジニア観点

- [ ] 全伝票（見積以外も含む）のステータスが列挙されているか
- [ ] 承認履歴エンティティの保持属性（必須/任意含む）が定義されているか
- [ ] 「残数量・残金額」の計算式（何から何を引くか）が定義されているか
- [ ] 入金消込のマッチング方法（手動/自動）が定義されているか

#### フロントエンドエンジニア観点

- [ ] 承認操作後の画面遷移が定義されているか
- [ ] 承認進捗の表示場所と内容が定義されているか
- [ ] ダッシュボードの各指標の集計単位とクリック遷移先が定義されているか

#### UXデザイナー観点

- [ ] 全フロー図に承認・差戻し・却下の3方向の経路が描かれているか
- [ ] システム操作とシステム外業務行為がフロー図上で区別されているか
- [ ] 外部アクター（顧客・仕入先）の応答をシステムに反映する経路が補記されているか

### 成果物

`/code-review/workflow-requirements-review-yyyy-mm-dd.md`

### テンプレート構成

```
## エグゼクティブサマリー（重大度別件数表）
## PM観点：業務要件の完全性
## TL観点：技術実装可能性
## BE観点：データモデル・ステータス遷移
## FE観点：画面遷移・UI
## UX観点：フロー図の抜け漏れ
## 総合評価と推奨アクション
  - 実装を開始できないまま放置すると発生するリスク
  - 推奨アクション（優先順位付き）
  - 実装開始可能と判断できる要件（ポジティブリスト）
```

### 注意点

- 「要件が曖昧」と「要件が未記載」は別物として区別して記録する
- 設計判断が必要なもの（ハードコードOK or 設定可能エンジン等）はPM・TLの合意事項として明示する
- 実装開始できる要件も明記することで、並行して進められる開発を止めない

##  5：指摘事項の修正案検討プロセス

### 目的

レビュー・検証で洗い出した指摘事項を、チームで1件ずつ議論して修正方針を確定させる。選択肢を明示して意思決定を促し、議事録に検討の経緯と理由を残すことで、後から「なぜこう決めたか」を追跡できる状態にする。

### 担当

テックリード（案の提示・推奨）、プロダクトマネジャー（意思決定）

### 手順

#### フェーズ 1：1件ずつ案を提示する

1. 指摘事項を重大度の高い順に1件取り上げる
2. 解決策の選択肢を **2〜3案** 提示する。各案に以下を記述する
   - 概要（一行）
   - 長所
   - 短所
3. テックリードとして **推奨案を1つ明示** し、推奨理由を述べる
4. PMへの **確認事項**（決定に必要な業務判断）を末尾にまとめる
5. PMの回答を待ち、次の件に進む

#### フェーズ 2：決定をレビューファイルに追記する

指摘事項の元ファイル（`workflow-requirements-review-yyyy-mm-dd.md` 等）の該当箇所に、以下の形式で決定を引用ブロックとして追記する。

```markdown
> **✅ 決定（yyyy-mm-dd）**: 採用した案と具体的な内容。
> 要件定義修正が発生する場合はその内容も記述する。
> **⚠️ 未決事項**: 継続検討が必要な項目があれば記述する。
```

#### フェーズ 3：議事録ファイルに記録する

`discussion-minutes-yyyy-mm-dd.md` に以下の構成で各件を記録する。

```markdown
## XX-NN：指摘事項タイトル

### 問題
（何が問題で、なぜ実装できないのかを簡潔に）

### 検討した案
（表形式で案・長所・短所を並べる）

### 決定
（採用した案と、その案を妥当と判断した理由）

### 要件定義修正
（修正内容を明記する）
```

末尾に **アクション項目一覧**（担当・状態）を更新する。

### ファイル運用ルール

| ファイル | 役割 | 更新タイミング |
|---------|------|-------------|
| `workflow-requirements-review-yyyy-mm-dd.md` | 指摘事項の一覧と決定の追記 | 各件決定のたびに即時追記 |
| `discussion-minutes-yyyy-mm-dd.md` | 検討経緯・選択肢・理由の記録 | 各件決定のたびに即時追記 |
| `requirements_definition.md` | 要件定義本体の修正 | 全件確定後にまとめて反映（または随時） |

### 案の提示フォーマット

選択肢が多いと議論が散漫になるため、**原則2〜3案に絞る**。

| 案 | 概要 | 長所 | 短所 |
|----|------|------|------|
| 案A | … | … | … |
| **案B（推奨）** | … | **…** | … |
| 案C | … | … | … |

### 注意点

- 「推奨案なし」で全案を並べて意思決定を委ねるのは避ける。推奨を明示することでPMの判断コストを下げる
- 推奨理由には「なぜ他の案より優れているか」を必ず書く。単に「シンプル」「要件と整合」では不十分
- 確認事項は「はい/いいえ」または「AかBか」で答えられる形式にする
- 1件の合意が取れてから次の件に進む（複数件をまとめて提示しない）
- 決定後の追記は即時行う。セッションをまたいでも議論の経緯が復元できるようにする
