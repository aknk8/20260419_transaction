## チームメンバーの構成

- プロダクトマネジャー：できあがったシステムに最終的な責任を持つ
- テックリード：エンジニアの監督
- UXデザイナー：画面のUXを企画、テストする
- フロントエンドエンジニアA：フロントエンドの実装
- フロントエンドエンジニアB：フロントエンドの実装
- バックエンドエンジニアC：バックエンドの実装
- バックエンドエンジニアD：バックエンドの実装
- QAエンジニアE：E2Eテストの設計、作成、実行
- QAエンジニアF：E2Eテストの設計、作成、実行

## ルール

- 開発者はTDDを採用する
- 実装計画に沿って、1ストーリー単位で実装し人間確認をはさむ、というステップで進める
- 人間確認の内容はユニットテスト通過と動作確認。E2Eテストの実行はストーリー完成時のみ行う
- E2EテストはPlaywrightを使う
- E2Eテストの結果はプロダクトマネジャーがレビューする

## テスト戦略

### テスト階層と実行タイミング
- **ユニット/統合テスト**（Vitest/Jest）: TDDサイクルごとに実行する。高速フィードバックを優先する。
- **E2Eテスト**（Playwright）: ストーリー完成時・PR作成前・リリース前にのみ実行する。TDDサイクルごとには実行しない。

## TDD (テスト駆動開発) ルール

### 基本原則
- **テストファースト**: すべての実装はテストを先に書いてから行う
- **Red-Green-Refactor**: このサイクルを厳密に守る
- **1テスト1実装**: 一度に1つのテストだけを追加し、それを通す実装を書く

### コーディング規約
- 関数の責務は1つに絞り、副作用を最小限に抑えること。
- 命名は直感的かつ記述的に行い、`bool` 値を返す関数は `is`, `has`, `can` などのプレフィックスを付ける。
- コメントで「なぜ」を説明し、「何」はコード自体で語ること。

### テスト命名規則
- describe: 対象の関数名またはクラス名
- it/test: `should [期待する振る舞い] when [条件]`
- 例: `it('should return 0 when input is empty string')`

### AAAパターン（必須）
すべてのテストは以下の構造に従う：

```ts
// Arrange（準備）
const input = createTestInput()

// Act（実行）
const result = targetFunction(input)

// Assert（検証）
expect(result).toBe(expected)
```

### 禁止事項
- テストを書く前に実装コードを書くこと
- テストを修正して実装に合わせること（実装を修正せよ）
- `test.skip()` や `test.todo()` を残したままにすること
- テストファイルで `any` 型を使うこと
- 1つのテストで複数の振る舞いを検証すること
- 実装の内部詳細に依存するテストを書くこと

### カバレッジ要件
- 新規コード: 80%以上（行カバレッジ）
- クリティカルパス（決済、認証）: 95%以上
- カバレッジが低下するPRはマージしない

### モック方針
- 外部APIのみモック可（内部モジュールは実際のコードを使う）
- Dependency Injectionパターンを優先
- モック対象は明示的にコメントで理由を記述
