# 探索的テスト計画書

**作成日**: 2026-05-04  
**対象**: 取引管理システム（現フェーズ実装済み範囲）  
**参加者**: プロダクトマネジャー、テックリード、UXデザイナー、フロントエンドエンジニア、バックエンドエンジニア、QAエンジニア

---

## 1. チーム議論の記録

### 1.1 探索的テストを実施する目的（PM発言）

> 「Playwrightで書いたE2Eテストはハッピーパスをカバーしている。でも、それだけでは安心できない。実際に画面を触ってみると、仕様に書いていなかった問題や、ユーザーが戸惑う操作フローが見つかることが多い。今回は静的レビューで『割引の解釈が違う』『ORD-00002のデータが壊れている』といった問題がすでに見つかっている。実際の画面でどう見えるか、どこまで影響が出るかを目で確認したい。」

**合意**: 探索的テストはスクリプトテストの補完であり、「まだ見えていないリスクを発見する」ことが目的。

---

### 1.2 テスト手法の選定（QAエンジニア提案）

> 「探索的テストにはいくつかアプローチがあります。今回はチャーターベース・セッションテスト（SBTM）を採用しましょう。チャーターとは『何を・どんな観点で・どこまで探索するか』を短く書いた指示書です。1セッションは60〜90分に時間を区切り、終わったら気づきを書き留めます。フリーハンドに触るより、チャーターがあると発見を共有・再現しやすくなります。」

**合意**: チャーターベース、1セッション60〜90分、セッション後に気づきシートを記入。

---

### 1.3 テストの優先領域（テックリード発言）

> 「静的コードレビューで見つかったリスクをそのまま探索テストの入力にするべきです。特に、割引の絶対額と割合の混在は実際の画面操作で数値がおかしくなるはずなので、見積→発注の一気通貫フローを必ず触ってほしい。また、S-07（納品・検収）がメニューから消えているのは、コードを見れば分かる話ですが、ユーザーとして触ると『発注したのに次に何をすればいいか分からない』という経験として現れます。」

---

### 1.4 UX観点の追加（UXデザイナー発言）

> 「私が見たいのは、権限が異なるユーザーで同じ操作をしたときの画面の差異です。admin、sales01、finance01で同じ画面を開いたとき、ボタンが消えたり、エラーになったりする挙動が一貫しているか。それから、フォームのバリデーションエラーのタイミングと文言も確認したいです。送信を押してから全部エラーが出るのか、入力中にリアルタイムで出るのかが画面によってバラバラだと利用者が混乱します。」

---

### 1.5 データ整合性の懸念（バックエンドエンジニア発言）

> 「ORD-00002のcustomerIdとQUO-00004のcustomerIdが違うというデータの問題は、見積一覧や受注詳細の画面で顧客名がどう表示されるかに直接影響します。また、見積の割引は円額なのに発注の割引は割合という問題は、受注から発注を起票するフローで金額がおかしくなるはずです。その境界を画面で確認してほしいです。」

---

### 1.6 フロントエンド固有の確認事項（フロントエンドエンジニア発言）

> 「商品マスタのunitPriceが文字列型で入っているのが気になっています。見積明細に商品を選択したときに単価が自動セットされますが、そこで型変換が起きているかどうかは画面で確認しないと分かりません。あと、ページングは5件区切りなので、マスタが多くなってきたときのページネーション操作が期待通り動くか確認したいです。」

---

### 1.7 セッション記録の方法（QAエンジニア補足）

> 「セッションが終わったら、以下の3点を書き留めてもらいます。(1)発見した問題と再現手順、(2)疑問に思った挙動、(3)期待どおりに動いた確認済みの範囲。完璧なバグレポートでなくてよいです。『何か変だと思った』レベルでもメモしてください。後でチームで仕分けします。」

---

### 1.8 対象外の合意（PM発言）

> 「今回の探索的テストはUI操作による確認を中心とします。パフォーマンス計測、セキュリティペネトレーション、DBスキーマの検証は対象外とします。データの永続化がない（リロードでリセット）ことは現フェーズの仕様として許容します。」

---

## 2. 探索的テストの進め方

### 2.1 基本フォーマット

```
【チャーター名】探索する対象と観点の一文
【担当】実施者名
【時間】XX分
【環境】http://localhost:5173（npm run dev）
【使用ユーザー】admin / sales01 / finance01 のいずれか（チャーターごとに指定）
【発見記録】セッション後にこのファイルの「3. 発見記録」欄に追記
```

---

### 2.2 優先度の考え方

| 優先度 | 基準 |
|---|---|
| P1（必須） | 基本業務フローが壊れる、データが消える・化ける、操作不能になる |
| P2（重要） | 権限制御の欠落、金額計算の誤り、画面遷移の欠落 |
| P3（改善） | 操作感の違和感、表示のちぐはぐ、エラーメッセージの不親切さ |

## 3. テストチャーター一覧

### チャーター ET-01 ── ログインと権限制御の全体探索
**目的**: 3種類のユーザーでログインし、メニュー・ボタン・画面アクセスの権限制御が一貫しているか確認する

**テスト対象**: ログイン画面 / サイドメニュー / 各画面の操作ボタン

**担当**: UXデザイナー

**時間**: 60分

**手順の指針**:
1. admin でログインし、表示されるメニュー項目をすべて記録する
2. sales01 でログインし、admin と比較して何が消えているか確認する
3. finance01 でログインし、同様に確認する
4. sales01 でマスタ管理画面（S-11）に直接URLアクセスを試みる（アクセス制御が機能するか）
5. finance01 で発注一覧に対して「新規発注」ボタンが表示されないことを確認する
6. ログアウト後にブラウザの戻るボタンで前の画面に戻れないか確認する

**注目ポイント**:
- メニュー項目がユーザー種別で正しく表示制御されているか
- 操作ボタン（編集・削除・承認依頼）が権限に応じて表示/非表示になっているか
- 直接URL遷移を試みたとき、ログイン画面にリダイレクトされるか

---

### チャーター ET-02 ── 見積作成から承認依頼までの一気通貫
**目的**: 見積の新規作成 → 明細入力 → 金額確認 → 承認依頼の流れで、業務的に矛盾のある挙動がないか確認する

**テスト対象**: S-04 見積一覧・登録・詳細

**担当**: バックエンドエンジニア

**時間**: 75分

**手順の指針**:
1. admin でログインし、「新規見積」を作成する（案件: PJ-00001、顧客: CUS-001）
2. 明細に商品を選択し、単価が自動セットされるか確認する（**unitPriceの型確認**）
3. 数量・値引きを入力し、小計・消費税・合計の計算が正しいか確認する
4. 値引きに金額（円）を入力したとき、計算式が `単価 × 数量 − 値引` になっているか確認する
5. 明細行を複数追加・削除し、合計が正しく再計算されるか確認する
6. 保存後に見積詳細画面で金額が一致しているか確認する
7. 「承認依頼」ボタンを押し、ステータスが「承認依頼中」に変わるか確認する
8. 既存の QUO-00001（承認済み）を改版し、版数が 2 になるか確認する

**注目ポイント**:
- 商品選択時の単価が数値として扱われているか（`"50000"` ではなく `50000`）
- 値引きの計算が見積と発注で同じ意味になっているか（コードレビュー 1-1 の確認）
- 改版時に旧版が保持されるか

---

### チャーター ET-03 ── 受注から発注起票への金額引き継ぎ確認
**目的**: 見積→受注→発注の伝票変換フローで金額が正しく引き継がれるか、特に割引の扱いを中心に確認する

**テスト対象**: S-05 受注詳細 / S-06 発注登録

**担当**: バックエンドエンジニア + テックリード（ペア）

**時間**: 75分

**手順の指針**:
1. 承認済みの見積 QUO-00001 から受注を作成し、金額が見積と一致するか確認する
2. ORD-00002 の詳細を開き、顧客名の表示を確認する（**データ不整合の影響確認**。QUO-00004 の顧客は CUS-005、受注の顧客は CUS-003 なので、どちらが表示されるか）
3. ORD-00001 から新規発注を起票し、発注金額が受注金額と一致するか確認する
4. 受注の明細に値引き（円額）が設定されている場合、発注の明細で値引きがどう解釈されるか確認する
5. 発注の「承認依頼」ボタンを押し、ステータスが「承認依頼中」に変わるか確認する

**注目ポイント**:
- ORD-00002 の顧客表示（コードレビュー 1-2 の影響）
- 見積→受注→発注の金額が一貫しているか
- 割引の意味が見積（絶対額）と発注（割合？）で化けていないか

---

### チャーター ET-04 ── マスタ管理の登録・編集・検索
**目的**: 顧客・仕入先・商品・ユーザーの登録・編集・検索・ページングが正しく動作するか確認する

**テスト対象**: S-11 マスタ管理（全タブ）

**担当**: フロントエンドエンジニア

**時間**: 60分

**手順の指針**:
1. 顧客マスタで新規顧客を登録し、一覧に追加されるか確認する
2. 顧客コードの自動採番が最大値+1（CUS-010）になるか確認する（既存 CUS-001〜009 が存在）
3. 顧客名をキーワードで検索し、部分一致で絞り込まれるか確認する
4. ページングを操作し（5件/ページ）、6件以上のデータで2ページ目に遷移できるか確認する
5. 既存顧客を編集し、変更が一覧に反映されるか確認する
6. ユーザーマスタで新規ユーザー（一般ユーザー）を登録し、権限がデフォルト設定になっているか確認する
7. 商品マスタで単価フィールドの入力値が数値として保存されるか確認する

**注目ポイント**:
- 採番が 3桁（CUS-010）で正しく動くか
- ページングが正しく機能するか
- 商品の単価が数値として扱われるか（コードレビュー 1-6）

---

### チャーター ET-05 ── 承認画面の操作と状態遷移
**目的**: 承認一覧（S-12）で、承認・差し戻しの操作がステータス変更に正しく反映されるか確認する

**テスト対象**: S-12 承認一覧 / 各伝票のステータス

**担当**: QAエンジニア

**時間**: 60分

**手順の指針**:
1. admin でログインし、承認一覧に見積・発注・支払の承認待ちが表示されるか確認する
2. 見積 QUO-00003（承認依頼中）を承認し、ステータスが「承認済み」に変わるか確認する
3. 承認後に見積詳細画面でステータスが更新されているか確認する
4. 別の見積を差し戻し、理由入力フィールドが表示されるか確認する
5. sales01 でログインし、承認一覧に「承認」ボタンが表示されない（または非活性）であることを確認する
6. finance01 で見積の承認操作が拒否されるか確認する

**注目ポイント**:
- 承認権限（`approval:act`）のないユーザーが承認操作できないか
- 差し戻し理由が保存されているか
- ステータス変更後に関連画面で反映されているか

### チャーター ET-06 ── 請求・入金画面の操作確認
**目的**: 請求一覧（S-08）・入金登録（S-09）の画面が正しく動作し、受注との紐付きが追跡できるか確認する

**テスト対象**: S-08 請求一覧・登録 / S-09 入金登録

**担当**: UXデザイナー + QAエンジニア（ペア）

**時間**: 75分

**手順の指針**:
1. finance01 でログインし、請求一覧（S-08）を開く
2. 請求対象の受注を選択し、請求を作成する
3. 既存の INV-00001（送付済）の詳細を開き、受注・顧客との紐付きが正しいか確認する
4. 入金登録（S-09）で INV-00001 に対して入金を登録しようとする（実装済みか確認）
5. 入金登録できた場合、残高の変化を確認する
6. 同一受注に対して2回目の請求作成を試みる（分割請求の可否を確認：コードレビュー 1-10）

**注目ポイント**:
- 分割請求が UI から操作できるか（現実装では同一受注コードで2回目の請求が除外される）
- 入金登録の UI が実装されているか
- 顧客・受注との紐付きが画面で確認できるか

### チャーター ET-07 ── エラー入力とバリデーションの振る舞い
**目的**: フォームの各種エラー条件でバリデーションが適切に機能するか確認する

**テスト対象**: 全登録・編集フォーム

**担当**: フロントエンドエンジニア

**時間**: 60分

**手順の指針**:
1. 顧客登録フォームで必須項目（顧客名）を空のまま保存し、エラーが表示されるか確認する
2. 顧客コードに既存コード（CUS-001）を入力して保存し、重複エラーが表示されるか確認する
3. 見積明細の数量に「0」「-1」「abc」を入力したときの挙動を確認する
4. 見積明細の単価に非常に大きな値（9999999999）を入力したときの合計計算を確認する
5. 日付フィールドに不正な形式（20260501 ← ハイフンなし）を入力したときの挙動を確認する
6. 複数フィールドでエラーが同時発生したとき、すべてのエラーが表示されるか確認する

**注目ポイント**:
- バリデーションのタイミング（保存時か入力中か）がフォームによって異なるか
- エラーメッセージが日本語で分かりやすいか
- 数値フィールドで文字列が入力できてしまわないか

## 7. 探索テストと既存テストの役割分担

| テスト種別 | ツール | 目的 |
|---|---|---|
| ユニットテスト | Vitest | ビジネスロジック単体の正確性 |
| E2Eテスト | Playwright | ハッピーパスの自動回帰確認 |
| **探索的テスト（本計画）** | 手動（ブラウザ） | 未知のリスク発見・UX確認・データ整合性確認 |

探索テストで発見した問題のうち再発リスクが高いものは、発見後に Playwright テストを追加して自動化する。
