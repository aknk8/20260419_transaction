# 取引管理システム 仕様書

## 1. 文書概要

### 1.1 文書名
取引管理システム 仕様書

### 1.2 作成目的
企業における以下の業務を一元管理するシステムの初期仕様を整理する。

受注業務
- 見積
- 受注
- 請求

調達業務
- 発注
- 支払い

本書は、関係者間で対象業務・必要機能・運用条件を明確化することを目的とする。

### 1.3 想定利用者
- 経営者
- 営業担当
- 営業事務
- 購買担当
- 経理担当
- システム管理者

### 1.4 対象範囲
本システムの対象は、顧客向けの見積作成から受注管理、仕入先への発注、顧客への請求、仕入先への支払い管理とする。また、取引先別の売上・費用、プロジェクト別の売上・費用の集計、ダッシュボード提供等管理会計の機能も担う。


以下は対象外とする。

- 会計システム本体の代替
- 在庫管理の高度な最適化
- 生産管理
- 倉庫管理
- 電子帳簿保存法、インボイス制度等への完全準拠実装
- 銀行振込の自動実行

## 2. システム概要

### 2.1 システムコンセプト
営業、購買、経理の各部門で分散管理されがちな取引情報を案件単位または受発注明細単位で統合し、業務の抜け漏れ防止、進捗可視化、証憑管理、承認統制、集計効率化を実現する。

### 2.2 解決したい課題
- Excel やメールで管理しており、見積・受注・発注・請求・支払いのつながりが追跡しにくい
- 二重入力が多く、転記ミスや請求漏れ、支払い漏れが発生する
- 承認フローが属人化し、内部統制が弱い
- 月次締め時の売上・原価・未収・未払の確認に時間がかかる
- 部門横断で取引状況を確認しにくい

### 2.3 導入目的
- 取引ライフサイクルの一元管理
- 業務標準化
- 承認履歴と操作履歴の記録
- 予実、売上、粗利、回収、支払い状況の可視化

## 3. 業務スコープ

### 3.1 管理対象プロセス
1. 取引先マスタ登録
2. 商品・サービスマスタ登録
3. 見積作成、見積承認、見積提出
4. 受注登録、受注承認
5. 発注計画、発注登録、発注承認、発注書発行
6. 納品、検収の記録
7. 請求書作成、請求確定、請求書発行
8. 入金確認
9. 支払依頼、支払確定、支払実績登録
10. 案件別、取引先別、期間別の集計と分析

### 3.2 基本管理単位
初版では以下の二つの管理単位を持つ。

- 案件
  - 顧客との商談や契約全体を管理する単位
- 取引伝票
  - 見積書、受注、発注書、請求書、支払依頼などの個別伝票

案件を起点に複数の見積、受注、発注、請求、支払いを関連付けられる構造とする。

## 5. 業務要件

### 5.1 見積管理

#### 5.1.1 目的
顧客への見積提出履歴を管理し、承認済み見積から受注へ連携できるようにする。

#### 5.1.2 要件
- 見積書を案件または顧客に紐付けて作成できる
- 明細単位で商品、数量、単価、値引、税区分を管理できる
- 見積番号を自動採番できる
- 見積の版管理ができる
- 下書き、承認依頼中、承認済み、失注、取消などの状態を持てる
- PDF 形式で見積書を出力できる
- 承認済み見積から受注を作成できる

#### 5.1.3 管理項目
- 見積番号
- 案件番号
- 顧客名
- 顧客担当者
- 見積日
- 有効期限
- 通貨
- 税区分
- 支払条件
- 納期
- 件名
- 備考
- 明細

### 5.2 受注管理

#### 5.2.1 目的
受注内容を確定し、売上予定や発注業務へ連携する。

#### 5.2.2 要件
- 見積から受注へ転記できる
- 受注番号を自動採番できる
- 受注ステータスを管理できる
- 受注金額、粗利見込、納期、納品状況を把握できる
- 一部受注、一部納品、一部請求に対応できる
- 受注から必要な発注を起票できる

#### 5.2.3 ステータス例
- 下書き
- 確定
- 手配中
- 納品中
- 納品完了
- 請求中
- 請求完了
- クローズ
- キャンセル

### 5.3 発注管理

#### 5.3.1 目的
受注内容に基づく仕入や外注を管理し、原価把握と支払い処理につなげる。

#### 5.3.2 要件
- 受注や案件から発注を作成できる
- 発注先を複数管理できる
- 発注明細ごとに納期、単価、数量、税区分を管理できる
- 発注番号を自動採番できる
- 発注承認フローを設定できる
- 発注書を PDF 出力できる
- 納品予定、納品実績、検収結果を記録できる
- 発注残を管理できる

#### 5.3.3 発注関連管理項目
- 発注番号
- 発注日
- 仕入先
- 支払条件
- 納品先
- 案件番号
- 関連受注番号
- 発注担当者
- 承認者
- 明細

### 5.4 請求管理

#### 5.4.1 目的
受注および納品実績に基づき顧客への請求を管理する。

#### 5.4.2 要件
- 受注または納品実績から請求データを作成できる
- 締め日、支払条件に応じた請求書作成ができる
- 複数受注を合算して請求できる
- 一部請求、分割請求に対応できる
- 請求番号を自動採番できる
- 請求書 PDF を出力できる
- 請求確定後は変更制御できる
- 売掛残高を管理できる

#### 5.4.3 ステータス例
- 下書き
- 確定待ち
- 確定
- 発行済み
- 一部入金
- 入金済み
- 滞留

### 5.5 支払い管理

#### 5.5.1 目的
仕入先への支払予定と支払実績を管理し、未払管理を行う。

#### 5.5.2 要件
- 発注や仕入実績から支払依頼を作成できる
- 支払予定日、支払条件、支払方法を管理できる
- 複数発注をまとめて支払い対象化できる
- 一部支払い、分割支払いに対応できる
- 支払承認フローを設定できる
- 支払実績を登録できる
- 買掛残高や未払残高を管理できる

#### 5.5.3 支払方法例
- 銀行振込
- 口座振替
- 現金
- 手形
- その他

### 5.6 入金管理

#### 5.6.1 目的
請求に対する入金状況を管理し、未回収の把握を可能にする。

#### 5.6.2 要件
- 請求単位または請求明細単位で入金実績を登録できる
- 一部入金や過不足入金を記録できる
- 入金日、入金方法、入金額、手数料を管理できる
- 入金消込を実施できる
- 未収一覧を出力できる

## 6. 業務フロー

### 6.1 基本フロー
1. 顧客、商品、仕入先などのマスタを登録する
2. 営業担当が案件を作成する
3. 見積を作成し、必要に応じて承認依頼を行う
4. 顧客から承認を得た見積をもとに受注登録を行う
5. 必要に応じて仕入先へ発注する
6. 納品、検収を記録する
7. 受注内容や納品実績をもとに請求書を発行する
8. 顧客からの入金を登録し、請求に消込する
9. 仕入先への支払依頼を作成し、承認後に支払実績を登録する
10. 各案件の売上、原価、粗利、未収、未払を確認する

### 6.2 例外フロー
- 見積却下時は理由を記録する
- 受注キャンセル時は関連発注、請求、支払いへの影響確認を行う
- 発注後の数量変更時は差分管理を行う
- 請求確定後の訂正は、原則として訂正伝票または取消処理で対応する
- 入金不足や過入金時は差額理由を記録する

### 10.2 承認要件

- 承認、却下、差戻しの履歴を保持する
- 代理承認の可否は別途設定できる

## 11. 非機能要件

### 11.1 性能
- 一覧画面は通常利用時に 3 秒以内で表示することを目標とする
- CSV 出力は 1 万件程度まで実用的な時間で出力できること

### 11.2 可用性
- 平日業務時間内の安定稼働を前提とする
- 障害時にデータ復旧可能なバックアップ運用を行う

### 11.3 セキュリティ
- ユーザ認証を行う
- ロールベースの権限制御を行う
- 通信は暗号化する
- 操作ログを保存する
- 添付ファイルへのアクセス制御を行う

### 11.4 監査
- 誰がいつ何を登録、更新、承認したかを追跡できる
- 削除、訂正、承認差戻しなど重要操作は監査対象とする

### 11.5 拡張性
- 将来的に会計システムや電子契約システムと連携可能な構成とする
- API 連携を前提にした設計が望ましい

## 16. 付録: MVP 案

### 16.1 MVP の目的
最小限の機能で、見積から請求、支払いまでの基本業務を止めずに回せることを優先する。

### 16.2 MVP 機能
- 顧客、仕入先、商品マスタ
- 見積登録、見積 PDF 出力
- 受注登録
- 発注登録、発注書 PDF 出力
- 請求登録、請求書 PDF 出力
- 入金登録
- 支払登録
- 取引一覧検索
- 操作ログ

### 16.3 MVP で割り切ること
- 高度な承認分岐は行わない
- 会計連携は CSV 連携とする
- ダッシュボードは簡易版とする
- 通知はメールのみに限定する
